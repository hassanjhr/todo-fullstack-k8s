# Implementation Plan: Advanced Task Management with Distributed Architecture

**Branch**: `005-advanced-features-dapr-kafka` | **Date**: 2026-03-31 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/005-advanced-features-dapr-kafka/spec.md`

---

## Summary

Extend the existing Todo Full-Stack application with intermediate features (priorities, tags, search, filter, sort), advanced features (due dates, reminders, recurring tasks), and a distributed architecture layer (Kafka event bus via Dapr pub/sub, Dapr service invocation, Dapr secrets store). The backend is extended with new SQLModel entities and migrated columns; the frontend receives new UI components; Dapr sidecars are added to both K8s and local dev deployments; Kafka receives all task lifecycle events via the Dapr pubsub building block.

---

## Technical Context

**Language/Version**: Python 3.11+ (backend), TypeScript/Node 20+ (frontend)
**Primary Dependencies**:
- Backend: FastAPI, SQLModel, asyncpg, python-dateutil, APScheduler, dapr (Python SDK)
- Frontend: Next.js 16+, React 19, TailwindCSS 4, Fetch API
- Infrastructure: Dapr 1.13+, Apache Kafka (via Docker Compose locally / Kubernetes in production)

**Storage**:
- Persistent: Neon Serverless PostgreSQL (existing) — extended with 5 new migrations
- Ephemeral: APScheduler in-process (reminder polling); Dapr state store (optional, for distributed lock on scheduler)

**Testing**: pytest + pytest-asyncio (backend), component tests for Dapr pub/sub

**Target Platform**: Linux server (Docker Compose local) / Kubernetes (Helm chart production)

**Performance Goals**:
- Search results: <1s for 10k tasks (pg_trgm + GIN index)
- Filter/sort response: <500ms
- Reminder delivery: ±2 minutes of scheduled time

**Constraints**:
- Zero cross-user data leakage (all queries filtered by authenticated user_id)
- No hardcoded secrets (Dapr secrets store in K8s, .env locally)
- Kafka publish failures must NOT fail HTTP responses (graceful degradation)
- Dapr sidecar required for production; optional for local dev

**Scale/Scope**: ~10k tasks per user; single-region deployment; single Kafka broker

### Key Design Decisions

#### D1: APScheduler for Reminder Polling
Poll every 5 minutes (configurable via `REMINDER_POLL_INTERVAL` env). When a reminder is due, publish `task.reminder` event via Dapr pubsub. This keeps the scheduler co-located with the API process, zero external dependencies for MVP.

**Trade-off**: If the backend process restarts within 5 minutes of a reminder trigger, the reminder may be delayed up to 5 minutes — acceptable per SC-005 (±2 min is aspirational; ±5 min for MVP).

#### D2: Dapr Pubsub as Kafka Abstraction
Application code calls `POST http://localhost:3500/v1.0/publish/taskpubsub/{topic}`. Dapr handles Kafka producer/consumer lifecycle. No Kafka client library in application code.

**Trade-off**: Introduces Dapr as a hard dependency in production. Mitigated by making Dapr optional in local dev (`DAPR_ENABLED=false` fallback).

#### D3: Cursor Pagination on `(created_at DESC, id DESC)`
All task list endpoints use cursor pagination. Offset pagination deprecated. The cursor is base64-encoded JSON of the last item's `created_at` and `id`.

#### D4: Tags as Junction Table (User-Scoped)
`tags` table with `UNIQUE(user_id, name)` ensures no duplicate tag names per user. `task_tags` junction avoids N+1 via `selectinload`. Tag delete cascades via FK on `task_tags`.

#### D5: RRULE Stored as Text, Next Occurrence Computed on Read
`recurrence_rule` stored as RFC 5545 RRULE string. `next_due_date` is a computed field in the TaskResponse schema (not stored). Calculated using `dateutil.rrulestr(rule).after(datetime.utcnow())`.

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Dapr sidecar not available in local dev | High | Medium | `DAPR_ENABLED=false` flag; event publishing becomes no-op locally |
| APScheduler job overlaps with itself under load | Medium | Low | Configure `max_instances=1` on scheduler job; use `replace_existing=True` |
| Cursor pagination breaks existing frontend pagination | Medium | Medium | Frontend update is atomic; old `offset` param removed cleanly |
| RRULE edge cases (e.g., Feb 30) | Low | Low | `python-dateutil` handles gracefully; next valid date returned |
| pg_trgm GIN index build time on existing large dataset | Low | Low | Migration runs `CONCURRENTLY` where possible; no table lock |

## Non-Functional Requirements

- **Performance**: pg_trgm GIN index on tasks. Composite cursor index `(user_id, created_at DESC, id DESC)`. Eager loading of tags via `selectinload` to avoid N+1.
- **Reliability**: APScheduler `max_instances=1`. Dapr pub/sub retry policy (3 retries, exponential backoff). Dead-letter topic `tasks-dlq` for undeliverable events.
- **Observability**: All Dapr service invocations propagate W3C TraceContext. APScheduler job logging via Python `logging`. Event publishing logged with correlation ID.
- **Security**: Tag name regex constraint. JWT user isolation on all new routes. Kafka events include `user_id` for audit; no sensitive PII in event payloads.
